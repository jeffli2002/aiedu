用 Front-End Design Skill + Nano Banana Pro，我终于解决了自己的页面审美焦虑

如果你跟我一样，是一个能把产品逻辑跑通、功能闭环也没问题的人， 那你大概率也经历过同一种痛苦：

页面能用，但不好看；
结构是对的，但没有“感觉”。

过去一段时间，我用 Claude Code（后来也大量用 Codex）， 已经可以非常顺利地完成项目的核心逻辑实现： 路由、数据流、SEO 结构、内容生成，全部在掌控中。

但只要一走到前端设计和页面配图这一步，我就会明显卡住。 不是不会写代码，而是更深层的问题：

  • 页面风格到底对不对目标用户？
  • 颜色是不是太冷？
  • 图是不是“看起来就很 AI”？
  • 整体有没有设计感，还是像工程 Demo？

这是一种产品人特有的审美焦虑。

真正的转折点：我开始系统使用 Front-End Design Skill

真正的变化，来自我开始把 Front-End Design Skill 当成项目流程里一个正式的能力模块，而不是“试一试”的工具。

先说一个很现实的前提： 我知道很多读者朋友并不是用官方模型在跑 Claude Code。

  • 有人用 GLM-4.7
  • 有人用 M2.1
  • 也有人是多模型混合

这完全没问题。
只要你用的是 Claude Code 这个执行环境，Skill 都可以正常工作。

安装这件事，其实比你想象中简单

如果你用的是 Claude Code

npx skills-installer install @anthropics/claude-code/frontend-design --client claude-code

我自己在一些项目里，就是用国产模型驱动 Claude Code， 并不影响 Skill 的使用。

如果你用的是 Codex

npx skills-installer install @anthropics/claude-code/frontend-design --client codex

这条命令同时对 Codex CLI 和 Codex IDE Extension 生效。 实际演示中，我更多是在 IDE 里完成。

如果你用的是 Cursor

npx skills-installer install @anthropics/claude-code/frontend-design --local --client cursor
安装完成后，在 Vibe Coding 时，只需要强调一句： “使用 frontend-design skill 来完成前端设计工作。” 就可以直接召唤它。

真正让我惊喜的，是它和 Nano Banana Pro 的结合

光有前端设计逻辑，其实还不够。 我过去最头疼的，是页面里的配图：

  • Hero 图
  • 模块分隔图
  • 情绪与氛围图

直到我把 Nano Banana Pro 的生图能力， 直接集成进 Front-End Design Skill 的流程里。

从那一刻开始，我不再“挑图”，而是“生成我需要的图”。

一个真实的项目场景

以我现在做的项目为例：

  • 目标用户：青少年为主
  • 视觉方向：活泼、暖色调、有设计感但不幼稚

我并没有直接对 AI 说“给我一张图”， 而是先通过对话确认这些产品判断。

然后 Skill 自动完成几件事：

  1. 把用户画像转成视觉约束
  2. 把情绪描述转成图片风格
  3. 调用 Nano Banana Pro 生图
  4. 直接落到页面结构中

很多时候，一次生成就够用了。

我开始“定制”这个 Skill

后来我做了一件很产品经理的事： 我没有停留在默认能力上，而是开始定制。

  • 固定主色
  • 统一字体
  • 约束圆角、阴影
  • 统一按钮风格

然后把这些偏好，写进一个新的 Skill。

从那以后，模型每次出页面、出图， 都自动带着我的品牌调性。

我真正解决的，不是设计问题

回头看，我解决的并不是“不会设计”， 而是把：

设计 SOP + 审美判断 + 生图能力
变成了一套可以被反复执行的系统

对我来说，Frontend-Design Skill 不是一个工具， 而是一个被固化的产品能力。

如果你和我一样，能跑通逻辑，却总卡在“看起来不对劲”， 那我非常建议你试一次这个组合。

下一步，我也会把这套流程整理成 可复用的 Skill 模板。
